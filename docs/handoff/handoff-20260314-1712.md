## Context Summary

Building a research baseline for a series of Text-to-SQL papers. The project extends an existing 4-phase RAG pipeline (Pre-Retrieval → Schema Linking → Generation → Post-Generation) with:

1. A **Schema Knowledge Graph (SKG)** — a typed heterogeneous graph over DB schemas with 3 layers of edges (structural DDL-derived, semantic name/description-based, statistical SQL co-occurrence + value overlap) and PPR-based retrieval. This is the main novel research contribution aimed at Paper 2 in the series.
2. **LLM provider abstraction** (`llms/`) — `BaseLLM` ABC with `OpenAILLM` (covers every OpenAI-compatible endpoint via `base_url`) and `HuggingFaceLLM` (local checkpoints, lazy load).
3. **Embedding model abstraction** (`embeddings/`) — `BaseEmbeddingModel` ABC with `HuggingFaceEmbeddingModel` using plain `sentence-transformers` (no langchain dep).

The research positioning: prior work is either graph-structured (RAT-SQL, LGESQL — Wave 1) OR LLM-description enriched (CHESS — Wave 2) but never both. The novel contribution is the **intersection**: graph-structured retrieval WITH LLM-enriched node descriptions/synonyms as node embeddings, retrieved via Personalized PageRank.

## Key Decisions Made

| Decision | Rationale | Alternatives Considered |
|---|---|---|
| `OpenAILLM` covers Ollama/vLLM/Groq via `base_url` | All are OpenAI-compatible; no separate provider classes needed | Separate OllamaLLM, vLLMProvider classes (deleted) |
| `HuggingFaceLLM` lazy-loads model | GPU memory not consumed on import; model only loads on first generate() | Eager load in __init__ |
| `openai` moved from optional to core dep | NodeEnricher + mine_correction_data both need it; always-required | Keep as optional `[openai]` extra |
| `[correction]` extra now only adds `anthropic` | `openai` is already core; simpler install surface | Keep openai in correction extra (redundant) |
| Graph edges stored as adjacency list `dict[str, list[KGEdge]]` | O(1) neighbour lookup during PPR; no NetworkX dependency | NetworkX graph (heavier dep, not needed) |
| PPR via power iteration (not scipy) | Pure Python, no scipy dep, graphs are small (≤200 nodes per DB) | scipy.sparse.linalg.eigs, NetworkX pagerank |
| LLM enrichment batches 15 nodes/call | Balances API cost vs. latency; fits in context window cleanly | 1 node/call (too slow), 50/call (JSON parsing error risk) |
| `NodeType.DATABASE` node included in graph | Enables DB-level retrieval context; consistent traversal from any node | TABLE as root (loses DB-level grouping) |

## Important Notes

- **Research novelty framing**: The key claim is that prior work uses graph structure OR LLM descriptions but not both. The ablation table must isolate: (A) flat LLM descriptions only, (B) graph structure only (Layer 1+2a, no LLM), (C) full method. Design experiments to support this.
- **BM25 tokenizer bug (known, unfixed)**: `hybrid_retriever.py` tokenizes on whitespace only — `employee_department_id` won't match "employee" or "department". Quick fix: split on `_` + camelCase boundaries. High ROI, 2 lines. Defer to separate PR.
- **GraphRetriever returns same dict schema as HybridRetriever**: keys `{id, content, chunk, score, source}` — this is intentional for drop-in compatibility with `SchemaFilter.filter_and_format()`.
- **`openai` is now a core dependency** (moved from `[openai]` optional extra). Any env without an OpenAI key will fail at `OpenAILLM.__init__()` but only if that class is instantiated — importing `llms` is always safe.
- **`ollama.py` and `vllm.py` were deleted** — if anything in the codebase imports from `llms.ollama` or `llms.vllm`, it will break. Grep before running: `grep -r "from llms" src/ scripts/`.
- **NodeEnricher expects `BaseLLM` interface** — any LLMFactory-constructed provider works. The default system prompt in OpenAILLM is SQL-focused; NodeEnricher overrides it with its own enrichment system prompt.
- **Graph JSON files can be large** (~50–200MB for full Spider with embeddings) — keep them in `data/schema_graphs/` which is gitignored.
