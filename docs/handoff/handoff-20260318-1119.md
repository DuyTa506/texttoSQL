## Key Decisions Made

| Decision | Rationale | Alternatives |
|---|---|---|
| Hybrid wins EX despite lower F1 | Recall dominates — missing tables = wrong JOIN, unfixable by LLM | Graph with hub-boosting could close gap |
| Merge ≈ Graph not Hybrid | RRF adds Hybrid noise above 15-slot SchemaFilter cap, displacing Graph's correct low-score tables | Asymmetric RRF weighting (graph_weight > hybrid_weight) |
| `batch_size=32` = 32 parallel API calls | Not 32 questions per prompt — each call has 1 question + 1 schema context | N/A |
| Use `BaseEmbeddingModel.embed_one()` for GraphRetriever | Project already uses OpenAI embeddings; removing SentenceTransformer dependency | Keep SentenceTransformer as fallback |
| Three parallel benchmark runs | Saves ~3-4 hours vs sequential; requires unique run dir per mode | Sequential safer, not worth the time cost |

## Architecture Reminder

```
Per question:
  Phase 1: ValueScanner → QuestionDecomposer → QueryAugmentor  [mostly disabled]
  Phase 2: HybridRetriever / GraphRetriever → SchemaFilter      [top-15 chunks]
  Phase 3: SQLInference.agenerate()                             [async, gpt-5-nano]
  Phase 4: RetryLoop → CandidateSelector                        [disabled]
  Eval:    execution_accuracy, exact_match, schema_recall/precision
```

Key config flags (all in `configs/default.yaml`):
- `schema_graph.enabled` / `schema_graph.hybrid` → Graph / Merge mode
- `schema_graph.graph_path` → `data/schema_graphs/bird_dev_v3_rebuild.json`
- `pre_retrieval.value_scan.enabled`, `pre_retrieval.decomposition.enabled`
- `generation.mode` → `standard` / `cot_plan` / `divide_conquer`
- `generation.n_candidates` → multi-candidate sampling
- `post_generation.retry_loop.enabled`
