## Context Summary

This session implemented 8 targeted retrieval quality fixes for the `GraphRetriever`
(Schema Knowledge Graph PPR-based retriever), ran a full BIRD Dev benchmark
(1 534 examples, 11 databases) comparing HybridRetriever vs GraphRetriever v2 vs
Graph+Hybrid merge, and wrote a detailed analysis report.

The GraphRetriever is the novel contribution of this research project — it uses
Personalized PageRank over a typed heterogeneous Schema Knowledge Graph as a
drop-in replacement for the BM25+ChromaDB HybridRetriever in the 4-phase
Text-to-SQL pipeline.

## Pending Work (Next Iteration — v3)

Prioritized by expected impact, as documented in `docs/report.md` §9.3:

- [ ] **HIGH** — Table-level PPR seeds for FK columns
  - When a FK column is selected as a seed, also add its parent TABLE node as a seed
  - Target: `formula_1.races` (28× missed), `financial.account` (23×), `thrombosis_prediction.patient` (24×)
  - Expected: +3–5 pp recall
- [ ] **HIGH** — Per-DB adaptive `score_gap_ratio`
  - Increase `gap_ratio` to 4–5 for DBs with `n_tables ≥ 10`
  - Target: `formula_1` (99× `laptimes` over-retrieved), `codebase_community` (121× `comments`)
  - Expected: −50+ noisy examples
- [ ] **MEDIUM** — Lower `alpha` to 0.65 + increase `max_hops` to 3 for chain-depth DBs
  - `european_football_2` chain: `match → player_attributes → player → team` barely reached
  - Expected: +2 pp recall on multi-hop schemas
- [ ] **MEDIUM** — Layer-3 `CO_JOIN` edge weight: 0.80 → 0.50
  - `codebase_community` statistical edges make all tables indistinguishable
  - Expected: better precision on stat-dense DBs
- [ ] **LOW** — Expand synonym vocabulary with gpt-4o (10 synonyms/node vs current 4–8)
  - Helps vague queries: "school info", "player stats"
  - Expected: +1–2 pp recall on simple queries
- [ ] **FUTURE** — End-to-end pipeline evaluation (retrieval → SQL generation EA score)
  - Current benchmark is schema-linking only; need to validate F1 gains translate to SQL accuracy

**Target after v3:** GraphRetriever F1 ≥ 0.78, recall ≥ 0.95, precision ≥ 0.58, avg tokens ≤ 90

## Key Decisions Made

| Decision | Rationale | Alternatives Considered |
|---|---|---|
| FK bridge injection BEFORE gap pruning | Bridge nodes have low scores and would be immediately pruned if gap pruning ran first | Run bridge after pruning (tried, bridge nodes always got pruned) |
| `min_keep=2` in gap pruning | Always keep at least 2 nodes regardless of gap — prevents complete miss on tiny DBs | `min_keep=1` (too aggressive for small schemas) |
| Adaptive max_nodes = `max(6, db_table_count)` | Floor of 6 ensures we always have enough nodes for JOINs even on the smallest DBs | Floor of 3 (too small for 2-table queries) |
| Column score filtering backward compatible | HybridRetriever path unchanged — `col_scores` dict empty → all columns shown | Unified path (would break existing Hybrid behavior) |
| Per-DB PPR subgraph via `G.subgraph()` view | O(1), no data copy, live view of original graph | Full subgraph copy (expensive), or post-filter (still leaks mass) |

## Architecture Reminder

```
Per question (inference):
  Phase 1 (Pre-Retrieval): ValueScanner + QuestionDecomposer + QueryAugmentor
  Phase 2 (Schema Linking): GraphRetriever OR HybridRetriever → SchemaFilter
  Phase 3 (Generation): SQLInference (standard/cot_plan/divide_conquer)
  Phase 4 (Post-Generation): RetryLoop + CandidateSelector
```

`GraphRetriever` lives in `src/schema_graph/graph_retriever.py`.
It calls `SchemaGraph.retrieve()` from `src/schema_graph/graph_builder.py`.
The graph file is built offline with `scripts/build_schema_graph.py`.

### GraphRetriever v2 Call Chain
```
GraphRetriever.retrieve(query, db_id=db_id)
  → _embed(query)                          # SentenceTransformer
  → _extract_synonym_tokens(query)         # Fix 4: NL stopwords filtered
  → _adaptive_max_nodes(db_id)             # Fix 2
  → SchemaGraph.retrieve(q_emb, ...)       # Fix 1: returns (KGNode, float) tuples
      → per-DB G.subgraph()                # Fix 5
      → nx.pagerank(run_graph, ...)
      → return [(node, ppr_score), ...]    # Fix 1
  → _add_fk_bridge_tables(...)             # Fix 6 (BEFORE pruning)
  → _apply_score_gap_pruning(...)          # Fix 3
  → _nodes_to_dicts(nodes_with_scores)     # Fix 1: uses real scores
  → [if hybrid_retriever] _rrf_merge(graph_results, hybrid_results)  # Fix 8
```

## Important Notes

1. **Graph file required**: The enriched graph file `data/schema_graphs/bird_dev_enriched_v2.json`
   is NOT in the repo (too large). Must be built locally with `build_schema_graph.py --enrich`.

2. **Two Chroma dirs in use**:
   - `./data/chroma_db_bird_analysis` — default for `analyze_retrieval.py`
   - `./data/chroma_db_bird_hybrid_only` — used for parallel hybrid run (to avoid lock conflict)

3. **v1 results not available** as JSON — the v1 overall F1=0.672 was from a prior session and
   only the overall number was saved. Per-DB breakdown for v1 is not available.

4. **`_SyntheticChunk`** in `graph_retriever.py` is a minimal stand-in for `SchemaChunk` that
   satisfies `SchemaFilter`'s attribute reads. It has `__slots__` for performance.

5. **Column score filtering** (Fix 7) in `schema_filter.py` uses the `"score"` key from chunk
   dicts and the `chunk.chunk_type == "column"` check. Graph path populates `col_scores` dict;
   Hybrid path leaves it empty → falls back to showing all columns (backward compatible).

6. **BIRD Dev dataset path** hardcoded in `analyze_retrieval.py` as
   `./datasets/data/bird/dev_20240627`. Adjust if your local path differs.
